> 狀態：Draft
> 日期：2026-03-12
> 作者：Chief Jin

## 1. 概述

### 1.1 系列名稱（暫定）

**「打造 AI Agent Skills 框架：企業級系統的設計決策與教訓」**

### 1.2 背景

作者花了 22 個活躍開發日、190+ commits、200+ Claude Code sessions，在一個擁有 400+ 頁面、17 種元件、9 種版型的企業前端系統上，從零建立了一套 14 個結構化技能（Skills）的 AI Agent 框架。本系列將這段設計演化歷程轉化為可供業界參考的技術文章。

### 1.3 目標讀者

已經在使用 AI 輔助開發工具（Claude Code、Cursor、Copilot 等），但尚未嘗試撰寫結構化 Skills 的工程師。

### 1.4 敘事風格

**演化紀實式（主題體）**：每篇聚焦一個設計維度，從「最初的天真假設」走到「最終的成熟設計」，穿插真實踩坑案例佐證。篇與篇之間主題獨立，但系列整體有邏輯順序。

### 1.5 內容原則

- **不可提及**：公司名稱、專案代號
- **保留**：數字與技術結構的具體性（頁面數、元件數、版型數）
- **泛化**：所有專有名詞替換為業界通用語彙（如 MLayout → Master-Detail Layout、posting → 表單提交驗證）
- **程式碼**：以設計論述為主體，搭配精選的結構性範例（Skill 骨架、HARD-GATE 語法、frontmatter 格式），不做逐步操作教學
- **定位**：偏重「設計決策與教訓」而非操作指南

### 1.6 規模

13 篇（導論 1 篇 + 核心主題 11 篇 + 展望 1 篇），每篇約 2000-3000 字。

## 3. 各篇設計

### #1 導論：當 AI 遇上 400+ 頁面的企業系統

**核心問題**：為什麼把 AI 丟進大型企業系統，光靠 prompt engineering 是不夠的？

**演化弧**：
- **天真假設**：「AI 很聰明，給它看 codebase 就能寫出正確的頁面」
- **現實衝擊**：MVP 測試一致性僅 72/100，出現 3 個嚴重幻覺——AI 編造了不存在的 Props、混淆了版型的事件模型、自創命名規則
- **轉折**：問題不在 AI 的能力，而在缺乏結構化的任務指引——400+ 頁面、17 種元件、9 種版型的排列組合超出了任何單一 prompt 能承載的範圍
- **引出解法**：需要一套「技能框架」，把隱性的開發知識轉化為 AI 可消費的結構化指引

**關鍵數據**：系統規模（頁面/元件/版型數）、MVP 測試分數、幻覺類型分類、最終框架規模（14 Skills、22 天、190+ commits）

**讀者收穫**：判斷自己的專案是否到了需要 Skills 框架的臨界點

### #2 TDD for Documentation：用測試驅動設計 AI 技能

**核心問題**：怎麼知道你寫的 Skill 是有效的？

**演化弧**：
- **RED Phase**：先不寫任何 Skill，記錄 AI 在無指引下的自然行為——6 類預期失敗點：(1) Props 編造（使用不存在的元件屬性）(2) 版型混淆（套用錯誤的 Layout 事件模型）(3) 命名自創（發明不符慣例的變數名）(4) 事件遺漏（跳過隱含的遠端查詢流程）(5) 風格偏離（不符既有 codebase 的寫作慣例）(6) 過度推論（從模糊規格書猜測實作細節）
- **GREEN Phase**：針對每個失敗點寫最小化的 Skill 內容，再測試是否修正
- **REFACTOR Phase**：Skill 內容過於冗長（643 行 → 302 行），拆分子檔案、外部化到 Docs
- **教訓**：回歸測試不可少——修了 A 版型的 Skill 後，B 版型的行為可能退化

**結構性範例**：
- RED Phase 的觀察記錄格式（失敗類型 × 嚴重度）
- 一個 Skill 從 v1（冗長敘述）→ v2（精簡 + 子檔案）的 before/after 對比

**讀者收穫**：一套可複製的「先觀察失敗、再寫指引、再驗證」方法論

### #3 漸進式披露：讓 AI 只讀該讀的

**核心問題**：AI 的上下文窗口是有限的，怎麼在大型知識庫中控制它的認知負荷？

**演化弧**：
- **v1**：所有資訊塞在一個 SKILL.md 裡（643 行），AI 讀到後半段開始遺忘前半段
- **v2**：拆出子檔案（events、examples），但 AI 不知道何時該載入哪個
- **v3（成熟版）**：三層載入架構——Layer 1 SKILL.md（決策樹 + 邊界條件，<500 行）→ Layer 2 按需載入子檔案（進入特定 Phase 時）→ Layer 3 跨 Skill 搜索（grep 精準定位）
- **關鍵決策**：合併相關 Skill 到父 Skill 子目錄（DRY 原則 vs 獨立性的取捨）
- **Token 經濟學**：「Gotchas Only」原則——SKILL.md 只放差異對比（1-2 行），完整說明外部化

**延伸案例**：Styling Skill 的「無截圖推論版面意圖」設計——當 AI 無法看到畫面截圖時，如何用規則推論出正確的 grid 佈局？這是漸進式披露在非結構化資訊上的應用。

**結構性範例**：
- 三層載入架構的示意圖
- 統一引用語法（為什麼禁止行號引用）

**讀者收穫**：設計 AI 知識庫時的資訊分層策略

---

### #4 Skills vs Docs：知識該放哪裡的設計演進

**核心問題**：哪些知識該寫在 Skill 裡，哪些該放到外部文件？

**演化弧**：
- **Phase 1**：所有東西都寫在 Skill 裡——元件表、事件清單、命名規則、範例全部塞入
- **問題浮現**：多個 Skill 引用同一個元件表 → 更新一處忘記其他 → 跨 Skill 不一致 → AI 得到矛盾指令 → 幻覺
- **Phase 2（概念收斂）**：建立「快速判定規則」——移除此段落後，AI 只需 grep Docs 即可獲得等效資訊 → 應外部化
- **成熟模型**：Skills 放 Gotcha / Flow / Red Flag（行為指引）；Docs 放 Domain-WHAT / Domain-HOW / Domain-RULE（事實知識）
- **交叉引用機制**：雙向可追溯的引用系統

**結構性範例**：
- 職責分離判定表（留在 Skills ✅ vs 外部化 ❌）
- 一個實際案例：元件 Props 從 Skill 內嵌到 Docs 外部化的重構過程

**讀者收穫**：建立知識架構時的 Single Source of Truth 原則與實踐方法

---

### #5 HARD-GATE：從建議到不可跨越的行為閘門

**核心問題**：AI 很擅長「合理化跳過步驟」，怎麼設計出它無法繞過的檢查點？

**演化弧**：
- **v1（建議式）**：Skill 裡寫「建議先讀完規格書再開始生成」——AI 經常「認為自己已經理解」而跳過
- **轉折事件**：一個複雜頁面，AI 跳過風險分析直接生成，結果事件模型完全錯誤，重做成本遠大於分析成本
- **v2（HARD-GATE）**：改用絕對禁止語法 `<HARD-GATE>` 標籤，明確列出「禁止做什麼」而非「建議做什麼」
- **全面部署**：從 1 個 Skill 擴展到 5 個關鍵 Skill + 全局治理文件，一天內 6 個 commits 完成
- **Red Flags 配套**：每個 HARD-GATE 搭配「危險念頭清單」——列出 AI 常見的繞過理由，搭配「現實」欄說明為什麼這念頭是危險的
- **四層防線設計**：入口 → 開發 → 規劃 → 驗證——每層各自防守，任何一層攔住都能避免最終錯誤

**結構性範例**：
- HARD-GATE 語法結構（標籤 + 禁止清單 + 觸發條件）
- Red Flags 表格格式（「念頭」→「現實」的對照結構）
- 四層防線的攔截流程圖

**讀者收穫**：設計 AI 行為約束時，「禁止式」比「建議式」有效得多；以及如何建構多層防線

### #6 會話分離：結構性消滅幻覺

**核心問題**：為什麼同一個 AI 在對話後半段的品質會下降？怎麼用架構手段解決？

**演化弧**：
- **觀察**：AI 在長對話中累積大量上下文後，開始混淆不同階段的資訊——規格書分析的結論污染了程式碼生成
- **假設**：上下文壓力是幻覺的主因，而非 AI 能力不足
- **實驗（Session-Aware Planning）**：強制將「規劃」和「執行」分到不同會話窗口——規劃窗口產出計畫文件，執行窗口只讀計畫 + 規格書
- **進階拆分**：執行窗口內部再按 Phase 設置 STOP 點——複雜案例建議再開新窗口
- **計畫文件作為接力棒**：窗口之間唯一的狀態傳遞介質是存檔的計畫文件
- **成效**：最複雜版型在導入會話分離後，回歸測試通過率顯著提升

**結構性範例**：
- 三窗口模式示意圖（Planning → Executing → Triaging）
- 計畫文件的最小結構（「決策 Only」原則——從「自包含」瘦身 63%）
- STOP 點與窗口切換的觸發條件表

**讀者收穫**：用「分窗口」取代「更長的 prompt」來對抗上下文幻覺

### #7 Orchestrator：AI 的任務路由器

**核心問題**：當框架有 14 個 Skills，AI 怎麼知道當前任務該載入哪些？

**演化弧**：
- **問題**：Skills 數量增長後，AI 開始「選錯 Skill」或「跳過必要 Skill」
- **v1（隱式路由）**：每個 Skill 自己描述觸發條件，靠 AI 自行判斷——經常誤判
- **v2（Orchestrator 集中路由）**：建立入口 Skill，定義「任務信號 → 必載技能鏈」的明確路由表
- **路由表設計**：4 種任務類型 × 對應技能鏈
- **多層閘門**：Orchestrator 自身也有 HARD-GATE——禁止在未完成任務分類前就載入任何 domain Skill
- **15 個禁止念頭**：Orchestrator 的 Red Flags 是最長的，因為它是第一道防線

**結構性範例**：
- 路由表結構（任務信號 → 技能鏈 → 強制流程）
- Orchestrator 與下游 Skills 的調度流程圖

**讀者收穫**：當 Skills 生態系成長到一定規模，需要一個「meta-skill」來管理路由

### #8 規格書解碼器：從人類語言到程式碼鏈

**核心問題**：企業系統的需求規格書充滿隱含慣例和模糊語句，AI 怎麼正確翻譯？

**演化弧**：
- **問題根源**：規格書用自然語言表述，背後是多行程式碼的完整流程——AI 只實作了表面的 UI 聯動
- **v1（靠 AI 自行推論）**：產生大量「看起來對但語意錯」的程式碼——比完全錯誤更危險
- **v2（隱含慣例解碼表）**：建立「規格書語句 → 完整程式碼鏈」的翻譯對照，共 10 條核心慣例
- **信心標記系統**：三級信心——✅ 明確寫出、🔵 依慣例推論、❓ 無法判斷。部分欄位列入「禁止推論清單」
- **強制輸出**：信心統計 + 未確認項清單 + 推論摘要 + 資訊缺口，四段缺一不可

**結構性範例**：
- 隱含慣例解碼表（精選 2-3 條展示落差）
- 信心標記的三級定義與禁止推論清單的設計邏輯

**讀者收穫**：AI 處理領域文件時「看起來懂」和「真的懂」之間的鴻溝；用信心標記讓不確定性可見化

### #9 品質根因診斷：修 Code 之前先修生態系

**核心問題**：AI 生成的程式碼有問題時，修完 code 就夠了嗎？

**演化弧**：
- **常見反應**：發現錯誤 → 直接修 code → 下次同類型頁面又犯同樣的錯
- **頓悟**：code 的錯誤只是症狀，根因可能在五個不同層級——規格書含糊、AI 遇模糊卻猜測、Skills 之間矛盾、Docs 缺漏模式、上下文過長
- **五層根因分析模型**：逐層檢查，定位真正的根因層級
- **Dual Fix 原則**：每個根因修兩處——code（立即止血）+ 生態系（防止復發）
- **HARD-GATE**：禁止「只修 code 不診斷根因」
- **實例**：一次診斷發現 6 類生態系問題，修復影響 4 個 Skills 和 2 份 Docs

**結構性範例**：
- 五層根因分析的檢查表結構
- 一個 Dual Fix 案例的 before/after

**讀者收穫**：建立「問題不在 AI，問題在指引系統」的思維模型

### #10 回歸測試與驗證閘門：Skills 的品質保證

**核心問題**：怎麼確保 Skills 的修改不會產生副作用？怎麼確保 AI 的產出符合規格？

**演化弧（回歸測試）**：
- **問題**：修了 A 版型的 Skill，B 版型的行為退化——因為共用部分元件定義
- **Sub-mission 機制**：每個版型獨立回歸測試基線，每次 Skill 修改後重新生成並比對
- **迭代實況**：四輪回歸，baseline 多次重新生成，每次都暴露新的不一致

**演化弧（驗證閘門）**：
- **v1**：測試和驗證混在一起，職責不清
- **v2**：獨立 Skill，三階段結構——風險評估 → 規格書四維度對照（欄位/事件/邏輯/不應存在）→ 技術合規檢查
- **「不應存在」維度**：不只檢查「該有的有沒有」，還檢查「不該有的是不是被 AI 自己加上去了」
- **HARD-GATE**：「在腦中比對完畢」不算驗證——必須輸出可見的結構化報告

**結構性範例**：
- 回歸測試的 baseline 比對流程圖
- 驗證報告的四維度結構
- ⚠️ 偏差（比 ❌ 缺失更危險）的設計理由

**讀者收穫**：Skills 不是寫完就好——需要回歸測試防止副作用、需要結構化驗證防止假陽性

### #11 收斂審查：14 個 Skills 的系統性治理

**核心問題**：當 Skills 數量成長到一定規模，個別品質不代表整體品質——怎麼做系統級治理？

**演化弧**：
- **累積債務**：3 週密集開發後，全局一致性劣化——同一概念在不同 Skill 有不同描述
- **全量審查**：14 個 Skills 系統性 review，47 項發現
- **三階段收斂**：Critical + Quick Wins → Token 效率優化 → 品質強化——刻意分階段，避免一次改太多
- **架構重構**：「Gotchas Only」原則的徹底執行——4 個超標 Skill 瘦身，觸發回歸測試，12/13 通過
- **硬編碼消除**：清單裡的硬編碼計數改為引用式或移除
- **術語統一**：規格書模板經歷三代命名，最終統一涉及 8 個 commits 橫跨多類文件

**結構性範例**：
- 47 項發現的分類框架
- 三階段收斂的排序邏輯

**讀者收穫**：Skills 框架是活的生態系，需要定期系統性審查；「分階段收斂」比「一次全改」更安全

### #12 從個人工具到團隊基建：導入與推廣

**核心問題**：一個人建的框架，怎麼讓整個團隊用起來？

**演化弧**：
- **安裝體驗**：手動複製 → 腳本化安裝，100% 達成率
- **版本管理**：開發庫與發佈庫分離——開發庫含開發產物，發佈庫只含 Skills + Docs
- **版本偵測**：自動版本元資料 + 啟動時檢查
- **教育訓練**：完整 2 小時工作坊，含「刻意出錯」示範腳本
- **反饋迴路**：Issue Template + 提交反饋 Skill + 審核反饋 Skill——使用者的踩坑回流到框架改進
- **變更同步機制**：規格書模板變更時，如何確保所有依賴該模板的 Skills 和 Docs 同步更新？建立了靜態依賴表 + 專責同步 Skill
- **雙遠端同步**：CI 層面確保開發庫與發佈庫不交叉污染

**結構性範例**：
- 開發庫 vs 發佈庫的檔案差異圖
- 反饋迴路的完整流程

**讀者收穫**：技術框架的「最後一哩路」是安裝體驗、版本管理和反饋機制

## 4. 部落格技術規格

### 4.1 平台

- 框架：Astro 5.x，Content Collections（glob loader + Zod schema）
- 部署：Cloudflare Pages
- 內容格式：Markdown

### 4.2 Frontmatter 結構


### 4.3 系列導航機制

目前部落格無 series 欄位。採用以下方案：

- **系列識別**：共用 tag `ai-agent-skills-series`
- **系列索引頁**：建立一篇索引文章（或獨立頁面），列出全系列 13 篇的標題、摘要與連結，作為系列入口
- **文章內導航區塊**：每篇文章底部放置 Markdown 格式的系列導航，包含「上一篇 / 下一篇」連結 + 「回到系列目錄」連結
- **Schema 擴展**：暫不修改 content schema 新增 `series` 欄位——13 篇的規模用 tag + 手動導航即可管理，避免過度工程化。若未來有更多系列再考慮擴展

### 4.4 updatedDate 策略

系列文章發佈後若因後續篇章的內容而需修訂前面的文章，使用 `updatedDate` 欄位標記，並在文章開頭加註更新說明。
